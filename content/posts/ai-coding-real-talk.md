去年底开始正式在工作里用 AI 写代码，到现在差不多三个月了。网上铺天盖地的都是"AI 改变编程""程序员要失业"，我想从一个普通开发者的角度说说真实体验。

不吹不黑，就聊我亲身经历的东西。

## 先说结论：能用，但没那么神

Copilot 现在是我写代码的标配了，每天都在用。但你要问我效率提升了多少，我只能说——看情况。

网上那些"效率提升10倍"的说法，要么是吹的，要么他原来的效率就很低。我个人体感大概是在**特定场景下提升30%-50%**，但整体下来算上调试AI生成的代码花的时间，可能也就提升个20%左右。

这个数字不难看，但也没到"颠覆"的程度。

## 真正好用的场景

说几个AI辅助编程确实很爽的场景。

### 重复性代码：这是AI的主场

写一堆 CRUD 接口的时候是真爽。以前十个类似的接口要一个个敲，CRUD的模式大同小异但又各有细微差别，手写起来又累又容易出错。现在 Tab 一下就出来了，准确率还挺高。

举个例子，我最近有个项目需要给二十多个数据模型写增删改查接口。以前这活至少要干一天半，现在 Copilot + ChatGPT 配合着来，大半天就搞定了。而且生成的代码格式统一、命名规范，比我自己手写的还整洁。

### 正则表达式：AI的天赋技能

以前写正则我得上 StackOverflow 翻半天，还要用 regex101 调试半小时。现在注释写清楚需求，它直接给你生成，还附带解释。讲真这个体验确实很爽。

上周有个需求要匹配一种复杂的中文地址格式，包含省市区街道门牌号，还有各种可选字段。我写了句注释描述需求，Copilot 给了一个正则，我在 regex101 上一测——直接通过了全部测试用例。以前这种正则我得调一个小时。

### 写测试用例：节省了大量无聊时间

单元测试是另一个AI表现出色的场景。你把函数写好了，让AI帮你写各种边界情况的测试，它能想到好多你自己想不到的case。比如空值、超长字符串、特殊字符、并发场景……

我现在的流程是：自己先写几个核心的测试用例确定逻辑对，然后让AI补充边界case。测试覆盖率明显上去了，而且省了不少无聊的重复劳动。

### 学新框架：AI是最好的入门导师

这个可能很多人没想到。上个月项目要用一个我不太熟的库（react-query v5），以前学新东西得看文档、看教程、看博客，花个半天才能上手。

现在我直接问 ChatGPT："我要用 react-query v5 做数据缓存和乐观更新，给我一个最佳实践的demo。" 它不仅给了代码，还解释了为什么这么写、有什么坑要注意。剩下的细节边写边问就行了。

当然了，AI给的不一定全对，但至少帮你快速建立了一个框架，比自己从零摸索效率高太多了。

## 真正难受的场景

但AI辅助编程绝不是万能的。有些时候它会让你抓狂。

### 复杂业务逻辑：它只会瞎猜

一旦涉及到业务逻辑比较复杂的部分，它就开始瞎猜了。有几次它自信满满地给我生成了一大段代码，看着挺对的，跑起来全是 bug。

最坑的是那种**"看着对但逻辑错"的代码**。比如有一次它生成了一段权限校验的逻辑，条件判断写得很漂亮，变量名也合理。但仔细一看，`&&` 和 `||` 的优先级搞反了，导致在某个边界条件下会放行不该放行的请求。这种bug你不仔细看真看不出来，上线了就是安全事故。

调试这种"半对"的代码，有时候比自己从头写还费时间。因为你既要理解它的思路，又要找出它哪里偏了，相当于在review一个你不认识的人写的代码。

### 项目上下文：它是个"外包"

还有一种情况很烦——它不了解你的项目上下文。你用了什么框架、什么版本、什么约定、什么历史债务，它统统不知道。

我们项目有一个自定义的请求封装层，所有的API调用都要走这个封装。但AI每次生成的代码都是直接用 `fetch` 或者 `axios`，风格完全不一样。你要改回来还得花时间。

更坑的是，它经常推荐已经废弃的API。有一次它给我用了一个 React 的 `componentWillMount`——这东西在 React 17 就标记为不推荐了，React 18 里基本没人用了。我一个React 18的项目，它给我写 class component + legacy lifecycle，我真的会谢。

### 调试：AI基本帮不上忙

遇到诡异的线上bug时，AI基本排不上用场。

上个月有个问题：用户反馈在某些情况下页面白屏。我排查了两天，最后发现是一个第三方CDN在特定region返回了被压缩但Content-Encoding header却没有设置gzip的响应，导致浏览器解析失败。

这种涉及到网络环境、第三方服务、特定配置的问题，AI连复现环境都搭不起来，更别说帮你查了。你把错误信息丢给它，它给你列一堆"可能的原因"，然后你发现都不是。

## 最大的隐患：变懒了

这个才是我最想说的。用了 AI 之后，我明显感觉到自己在"偷懒"。

以前遇到不会的东西，我会去看文档、查源码、理解原理。现在的第一反应是"问 AI"，拿到答案就用，不去深究为什么。短期来看效率是提高了，但长期下来技术成长肯定会受影响。

### 一个让我警醒的瞬间

上个月有一次，同事问我一个关于 Promise 执行顺序的问题——`Promise.resolve().then()` 和 `setTimeout` 谁先执行、为什么。

我竟然愣了一下才反应过来。以前这种微任务队列和宏任务队列的基础知识是随口就答的，事件循环机制、microtask/macrotask的优先级，这些都是我面试时必问别人的题目。现在我自己都得想一想了。

说真的，这个趋势让我有点慌。

### 知识"空心化"的风险

我后来仔细想了想，发现问题不只是"变懒"这么简单。AI辅助编程可能导致一种**知识空心化**：你知道怎么用，但不知道为什么。

就像用计算器的人可能心算能力退化一样，长期依赖AI写代码可能让你的底层技术功力退化。代码是AI帮你写的，你读了觉得"嗯看起来对"就用了。但如果哪天AI不可用了（比如网络问题、服务宕机），你还写得出来吗？

更严重的是：如果你不理解代码的底层原理，你就没法在AI出错的时候发现错误。这会让你的代码质量下降，而你自己还不知道。

## 各工具横向对比

三个月里我用过几个主流的AI编程工具，简单说说感受：

| 工具 | 优点 | 缺点 | 适合场景 |
|------|------|------|----------|
| GitHub Copilot | 嵌入VS Code无缝，自动补全自然 | 对长文件理解有限 | 日常编码 |
| ChatGPT | 交互式对话，适合探索性问题 | 代码风格不统一 | 学习新技术、解决疑难杂症 |
| Claude | 长上下文理解好，代码质量高 | 不能联网搜索 | 代码审阅、重构 |
| Cursor | AI原生编辑器，体验流畅 | 有时过于aggressive | 需要频繁与AI交互的项目 |

我目前的组合是：**日常写码用 Copilot + 遇到问题开 Claude 聊 + 偶尔用 ChatGPT 搜索新东西**。

## 给同行的使用策略

折腾了三个月，总结出一套自己的方法：

### 1. 分层使用：不同类型的代码区别对待

- **模板代码** → 放心交给AI：CRUD、表单、测试、文档注释
- **业务逻辑** → 自己主导，AI辅助：先理清思路，再让AI帮你补全细节
- **核心算法** → 自己写：涉及到安全、金融、数据一致性的代码，自己动脑子
- **架构设计** → 完全自己来：AI不懂你的业务全貌

### 2. Code Review 是底线

不管AI生成的代码看起来多正确，**每一行都要理解**。不要无脑 Tab。

我现在养成了一个习惯：AI生成的代码，我会像review同事代码一样仔细看。如果有看不懂的地方，就问它"这行代码为什么这么写"，直到我完全理解为止。

### 3. 保持手感

每周至少花半天时间，关掉所有AI工具纯手写代码。就像运动员在有辅助训练设备的情况下，也需要barefoot跑步一样。

### 4. 对AI生成的代码保持怀疑

AI的自信和它的正确性之间没有任何关系。它可以用极其自信的语气给你一段完全错误的代码。**永远不要因为AI说得很有道理就不去验证。**

### 5. 记录你从AI那里学到的东西

遇到AI教你的好用技巧或者设计模式，记下来。不然下次你还是得问AI，知识就永远不是你的。

## 三个月后的心态变化

刚开始用的时候很兴奋："太牛了，以后不用写代码了！"

用了一个月开始冷静："嗯……好像也没那么牛。"

用了三个月回归理性："这就是个工具，跟IDE的自动补全本质一样，只是更智能了。"

AI助手不会让差程序员变好，也不会让好程序员变差。它只是让你在擅长的事情上更高效，在不擅长的事情上暴露得更快。

关键在于你怎么用它。刀能切菜也能伤手，道理都一样。
